# CARBON PROJECT DIRECTORY & SYSTEM ARCHITECTURE SPECIFICATION

This document defines the production-grade directory structure for the **Carbon** platform, ensuring strict adherence to System Design principles, modularity, and Security Quality Management (SQM).

## 1. ROOT DIRECTORY STRUCTURE
The root directory segregates the project into distinct environments to facilitate independent versioning, deployment, and scalability.

```markdown
carbon-project/
├── backend/                # FastAPI Micro-Architecture
├── frontend/               # Flutter (Mobile) Clean Architecture
├── scripts/                # Database migration and simulation scripts
├── docker-compose.yml      # Container orchestration (optional)
└── README.md               # Project documentation
```

## 2. BACKEND DIRECTORY (FASTAPI)
The backend follows a **Layered Pattern** to decouple business logic (Parametric Engine) from infrastructure (Database/API).

```markdown
backend/
├── app/
│   ├── api/                # Route controllers
│   │   ├── v1/
│   │   │   ├── worker.py   # Registration and profiles 
│   │   │   ├── policy.py   # Premium and coverage logic 
│   │   │   ├── payout.py   # Parametric trigger 
│   │   │   └── admin.py    # Disruption simulation and analytics 
│   ├── core/               # Cross-cutting concerns
│   │   ├── config.py       # Pydantic-based env management
│   │   ├── security.py     # HMAC signing and sensor fusion gates 
│   │   └── constants.py    # Multiplier matrices and thresholds
│   ├── db/                 # Persistence layer
│   │   ├── session.py      # Supabase/PostgreSQL session management
│   │   └── models.py       # SQLAlchemy ORM definitions
│   ├── schemas/            # Pydantic data validation (DTOs)
│   ├── services/           # Business Logic Layer
│   │   ├── weather_svc.py  # Rule-based Weather Synthesizer
│   │   ├── sensor_svc.py   # Kinematic analysis for fraud detection [cite: 500, 505]
│   │   └── alert_svc.py    # Firebase Cloud Messaging (FCM) push
│   └── main.py             # Entry point with lifespan events
├── tests/                  # SQM validation scripts
├── .env                    # SECURE: API keys and DB credentials
├── .gitignore              # Prevents .env from being committed
└── requirements.txt        # Dependency management
```

## 4. OPTIMIZATION & FLEXIBLE HANDLING TECHNIQUES

### 4.1 Secure Environment Management
To prevent credential leaks, sensitive items MUST reside in `.env` files. 
- **Backend:** Utilize `python-dotenv` and Pydantic `BaseSettings`. This ensures that if a required key (e.g., `SUPABASE_KEY`) is missing, the server fails to start, maintaining SQM.
- **Frontend:** Use `flutter_dotenv` to inject environment variables at runtime.

### 4.2 Path Aliasing
For the backend, implement path aliasing or relative imports within `sys.path` to ensure that moving the `services` folder does not break `api` routes. In Flutter, utilize package imports exclusively to avoid "relative import hell" and maintain modularity.

### 4.3 Database Hybrid Handling
- **Firebase (Hot Data):** Located in `frontend/lib/data/providers` for direct driver-to-driver location pings to minimize latency.
- **Supabase (Cold Data):** Accessed via the FastAPI backend to enforce business logic and audit logging before any financial state change.

## 5. CENTRALIZED MATERIAL THEME CONTROL (FRONTEND)
To ensure a uniform UI and rapid enhancement, the `lib/core/theme/` directory will house a singleton `AppTheme` class. This class will define:
- [cite_start]**Color Palettes:** High-contrast variants for sunlight visibility.
- **Component Themes:** Global styling for `st.metric` style cards, buttons, and input fields.
- **Typography:** Scalable text themes for readability during active delivery tasks.

*Note: Once you provide the specific theme control logic, it will be integrated into the `AppTheme` singleton to override the standard Material 3 defaults across the entire application.*

## 6. SQM VALIDATION PROTOCOL
Every component in the directory must be validated using the following logic:
1. **Directory Integrity:** Check that `models.py` only contains schema definitions and no business logic.
2. **Security Check:** Run `git status` to ensure `.env` files are ignored by `.gitignore`.
3. **Logic Verification:** Ensure that the `weather_svc.py` correctly implements the consistency filters (e.g., Rain > 5mm implies Humidity > 85%) before deployment.
